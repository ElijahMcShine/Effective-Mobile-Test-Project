## Chapter I
## Общая информация

Аккаунт на GitHub.com не является основным моим местом выкладывания проектов. Проекты, которые могу выкладывать публично находятся тут: https://gitlab.mcshine.ru/public-projects

Проект представляет собой несколько вариантов исполнения связки "backend webserver - nginx proxy".

Пояснения к работе каждого из них находятся в [Главе 3](#chapter-iii).

При запуске любого из них создаются и запускаются контейнер приложения (веб-сервера), контейнер прокси сервера и сеть, для их взаимодействия. В данном примере рассматривается связка "обратный прокси -> backend". Такой метод удобен по нескольким причинам: 
- один прокси может пересылать запросы ко многим серверам, что экономит количество внешних ip адресов
- по запросам одного доменного имени (по одному адресу узла), но по разным путям, прокси может перенаправлять запрос на разные сервисы
- можно настроить балансировку нагрузки

Ответы от сервера приложения не доступны вне сети взаимодействия этих контейнеров. Прокси сервер отвечает по порту 80.

В качестве ответа ожидается строка "Hello from Effective Mobile!", которая, для большего соответствия web стандарту обернута в html. В [Главе 3](#chapter-iii) указано, как убрать htlm.

Проект реализован средствами Docker контейнеризации, с применением Docker compose для удобства развертывания.

Типовая карта проекта, есть отличия по файлам исходного кода приложения:
src
└--ver(number)
    ├--docker-compose.yml    - Файл docker-compose для сборки определенной версии
    ├--python.env            - Опциональный файл для переменных окружения 
    ├--backend               - Каталог для сборки приложения
    |   ├--app.py            - Файлы исходного кода приложения
    |   ├--.dockerignore     - Файл для уменьшения контекста сборки приложения
    |   └--Dockerfile        - Докерфайл для приложения
    └--nginx                 - Каталог для сброки прокси
        ├--nginx.conf        - Файл конфигурации прокси
        ├--.dockerignore     - Файл для уменьшения контекста сборки прокси
        └--Dockerfile        - Докерфайл для прокси

## Chapter II
## Инструкция пользователя

### Запуск.

Для запуска любой версии выполните следующее. \
Вариант 1:
- В командной строке перейдите в директорию необходимой версии (src/ver{1..3})
- Выполните команду "sudo docker-compose up -d"

Будет произведена сборка и запуск приложения. Однако, если вы произвели изменения в версии, то перед запуском стоит пересобрать образы командой "sudo docker-compose build"

Вариант 2:
- Выполните команду запуска с указанием пути к файлу docker-compose.yml. Например, находясь в директории корня проекта введите "sudo docker-compose -f ./src/ver1/docker-compose.yml up -d"

Если же были сделаны изменения в версии, то пересобрать образы можно так же с указанием файла docker-compose.yml: "sudo docker-compose -f ./src/ver1/docker-compose.yml build"

### Удаление.

Выполните команду sudo docker-compose down из директории версии и docker-compose выполнит последовательную остановку контейнеров (чего можно добиться командой docker-compose stop) и команду удаления (docker-compose rm).

Для удаления также можно использовать указание файла docker-compose.yml.

### Проверка работоспособности.

Стандартная команда "sudo docker ps -a" должна показать запущенные и работающие контейнеры. Примерно оно выглядит так:
```
CONTAINER ID   IMAGE                 COMMAND                  CREATED              STATUS                                 PORTS                                         NAMES
8c05ce92114d   ver3_ver3-webserver   "../docker-entrypoin…"   About a minute ago   Up About a minute (health: starting)   8080/tcp                              ver3-webserver
5f54152beee8   ver3_ver3-proxy       "../docker-entrypoin…"   About a minute ago   Up About a minute (healthy)            0.0.0.0:80->80/tcp, [::]:80->80/tcp           ver3-proxy
```

Если статус не "Up", то попробуйте удалить и запустить заново.

Также работоспособность выражается в получении ожидаемого ответа на команду "curl localhost", введенную на сервере, на котором и развернуто приложение, или укажите адрес сервера, если запускаете на другом компьютере.

Или же воспользуйтесь веб-браузером. Если таковой имеется на сервере, то в строке url укажите localhost, или, если запускаете на другом компьютере, укажите адрес сервера.

## Chapter III
## Описание варианта 1

Данный вариант базируется на использовании Python как веб-сервера с использованием библиотеки http.server.

Ожидаемым ответом по умолчанию будет строка:
<html><head><title>Hello from Effective Mobile!</title></head><body><p>Hello from Effective Mobile!</p></body></html>

Если желаете изменить на указанную в задании строку без использования html измените код:
```
self.wfile.write(bytes("<html><head><title>Hello from Effective Mobile!</title></head>", "utf-8"))
self.wfile.write(bytes("<body><p>Hello from Effective Mobile!</p>", "utf-8"))
self.wfile.write(bytes("</body></html>", "utf-8"))
```
в файле src/ver1/backend/app.py на следующий:

```
self.wfile.write(bytes("Hello from Effective Mobile!", "utf-8"))
```

Nginx прокси указывает на имя контейнера приложения:
```
proxy_pass http://ver1-webserver:8080;
```

Поэтому, если хотите изменить, то изменяйте в связке.\
src/ver1/nginx/nginx.conf:
```
proxy_pass http://ver1-webserver:8080;
```
src/ver1/docker-compose.yml
```
services:
  ver1-webserver:
    container_name: ver1-webserver
```

Также в данном варианте продемонстрированы разные варианты указания HEALTHCHECK. Для прокси указано в src/ver1/nginx/dockerfile, а для сервиса приложения в src/ver1/docker-compose.yml.

## Описание варианта 2

Данный вариант демонстрирует приложение на fastcgi, написанное на языке С.

Для изменения ожидаемого ответа следует внести правки в следующие строки файла src/ver2/backend/mini_web.c:
```
printf("<!doctype><html><head><title>Hello from Effective Mobile!</title></head>\n");
printf("<body><p>Hello from Effective Mobile!</p>\n");
printf("</body></html>\n");
```

заменить на:
```
printf("Hello from Effective Mobile!\n");
```

Также данный вариант примечателен использованием многостадийной сборки. Сначала в одном образе устанавливаются необходимые для компиляции приложения библиотеки, а потом готовый исполняемый файл копируется в рабочий образ.

HEALTHCHECK реализован на наличии открытого порта 8080.

В части прокси изменение относительно первого варианта в проксировании fastcgi:
```
fastcgi_pass ver2-webserver:8080;
```

## Описание варианта 3

Третий вариант основывается на использовании nginx не только как обратного прокси, но и как веб-сервера.

Сама страница ответа представляет гипертекст в файле src/ver2/backend/index.html

По умолчанию nginx слушает порт 80, но дополнительно открыт порт 8080.

## Описание варианта 4

Четвертый вариант представляет собой измененный вариант 1.

Добавлен поиск ответа в базе данных. И выдача на основании выбора ответа из списка в БД.

Осознанно упрощен код приложения, не установленны проверки исключений, т.к. задача данного примера показать работу и логику зависимостей в docker-compose.
Пока не будет доступна база данных не будет развернут сервис приложения.

Используется баз данных Postgresql, файл модели БД, который автоматически разворачивает первоначальную базу данных, а также Python модуль psycopg2 для работы с БД.

## Chapter IV
## Изменения по замечаниям

1) Отсутствует .gitignore файл \
    Добавлен

2) Отсутствует .dockerignore файл (увеличивает размер контекста сборки) \
    Добавлено, хотя контексте данного задания его отсутствие не влияет на размер контекста сборки

3) Отсутствует .env файл для переменных окружения \
    Добавлен в первом и четвертом варианте исполнения

4) В nginx.conf отсутствует X-Forwarded-Proto заголовок \
    Список передаваемых заголовков стоит оценивать исходя из задач. \
    Статья на тему передачи заголовков https://fastfox.pro/blog/tutorials/reverse-proxy-header-hygiene/

5) Нет upstream блока в nginx.conf (проксирование напрямую на service:port) \
    Добавлено, хотя в контексте данного задания не влияет.

6) Нет restart policy в docker-compose.yml \
    Добавлено

7) Нет depends_on в docker-compose.yml \
    В контексте данного задания не имеет смысла использовать зависимости, так как прокси сервер может работать и без работающего приложения. \
    Работающее же приложение без работающего прокси просто не получит запросов, но на работе самого приложения это не отразится.

    Для демонстрации использования зависимостей был подготовлен четвертый вариант, в котором приложение обращается к базе данных для выбора ответа. \
    В таком случае приложение не может работать без работающей базы данных.

8) README не содержит ASCII схемы архитектуры \
    Добавлено

9) В ver2 nginx.conf используется proxy_set_header вместо fastcgi_param для FastCGI \
    Добавлено. Также надо заметить, что в первом и втором варианте добавлен вывод заголовков приложением, для демонстарации передачи. \
    В первом варианте приложение выводит в лог, во втором варианте приложение выводит в ответе.
